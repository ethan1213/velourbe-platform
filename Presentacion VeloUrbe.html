1 · Introducción

Contexto y Solución

VeloUrbe resuelve la movilidad urbana de última milla con un ecosistema digital que permite arrendar patinetas eléctricas de forma ágil y segura. Adopta una arquitectura de 10 microservicios tras un API Gateway y un BFF para desacoplar identidades, flota, pagos y la experiencia del cliente.

Problema

Movilidad de última milla: acceso rápido y seguro a patinetas en la ciudad.

Solución

10 microservicios de dominio tras un Gateway y un BFF, orquestados con Docker Compose.

Enfoque

Clean Code, patrón CSR, JWT stateless, Flyway y pruebas automatizadas.

Resultado

Sistema cohesivo, escalable y tolerante a fallos, listo para crecer.

Casos de Uso Principales

1Registro y Autenticación

El cliente envía credenciales al Gateway → pasan por el BFF al servicio de auth → se hashean y persisten → retorna un JWT que habilita el acceso a los módulos protegidos.

2Ciclo de Vida del Arriendo

Usuario autenticado inicia un viaje → se verifica disponibilidad transaccional → el scooter pasa a "en uso" → al finalizar se libera el recurso, se contabiliza el tiempo, se procesa el pago y se puede dejar una reseña.

3Orquestación del Dashboard

El cliente consulta al BFF → ejecuta llamadas concurrentes no bloqueantes vía WebClient a los servicios core → ensambla un payload unificado y optimizado.

2.3 · C4 Model

Diagrama C3 — Patrón CSR (Controller → Service → Repository)

Security · JwtAuthFilterValida firma del token y extrae claims (sub, role, userId). Sin token válido → 403.
↓
Controller LayerMapeo REST + HATEOAS · sin lógica de negocio ni repositorios
↓  DTOs inmutables (Java Records)  ↓
Service LayerReglas de negocio · @Transactional · único acceso al repositorio
↓  Entities  ↓
Repository · Spring Data JPAAcceso a datos · ORM Hibernate · solo queries
↓
Flyway → PostgreSQLMigraciones DDL versionadas (V1, V2…) · BD propia por servicio

Comunicación interna

  • Las capas intercambian DTOs inmutables (Java Records).
  • Las Entities solo viven entre Service y Repository.
  • Ningún controller inyecta repositorios — CSR verificado en los 10 servicios.

Comunicación externa

  • Capa Client (WebClient) presente en el BFF.
  • El BFF proxea /api/** y agrega datos de los 10 microservicios sobre HTTP.

3 · Base de Datos

Persistencia y Migraciones (Flyway)

Cada uno de los 10 microservicios tiene su propia BD PostgreSQL (incluidas db_scooter_stations y db_scooter_reviews). Flyway versiona el esquema con DDL separado del DML.

user-auth · db_scooter_users

V1 → tabla users (DDL)
V2 → seed admin (DML)
V3 → fix password

rental · db_scooter_rentals

V1 → tabla scooters (DDL)
V2 → tabla rentals (DDL)

station / review (nuevos)

V1 → tabla stations (DDL)
V1 → tabla reviews con UNIQUE (user, rental)

Separación estricta DDL / DML

Prohibido mezclar estructura (CREATE TABLE) con datos de prueba (INSERT) en un mismo archivo. Los datos de prueba se generan con DataFaker solo bajo el perfil dev, garantizando despliegues limpios en producción.

4 · Pruebas

Estrategia de Testing

Servicio

@ExtendWith(MockitoExtension)
Lógica de negocio aislada con dependencias simuladas, sin contexto Spring.

Controller

MockMvc standalone + Mockito
Mapeo HTTP, códigos de estado (201/200/404/401) y serialización JSON.

Contexto

@SpringBootTest + perfil test
El contexto levanta con H2 en memoria — la suite corre sin PostgreSQL.

user-auth-service

AuthServiceTest · UserServiceTest · AuthControllerTest · UserControllerTest
Registro, login, email duplicado, 401 inválido…

rental / payment / station / review

ScooterServiceTest · RentalServiceTest · StationServiceTest · ReviewServiceTest · PaymentControllerTest…
Arriendos, dock/undock, estación llena, rating promedio, reseña duplicada…

Cobertura total: 11 clases de test entre servicios y controladores.

5 · Documentación API

Swagger Centralizado + Postman

Swagger centralizado en el Gateway

http://localhost:8080/swagger-ui.html — un solo Swagger UI agrupa la documentación de los 10 microservicios + BFF mediante un selector de definiciones. El Gateway expone cada /v3/api-docs bajo rutas /docs/{servicio} usando solo configuración (SetPath) — sin código.

Swagger por servicio

Cada microservicio mantiene su propio UI:
localhost:8081…8091/swagger-ui.html

Colección Postman (/postman)

Endpoints por servicio + carpetas Stations, Reviews, BFF y Vía API Gateway, con entorno de variables preconfigurado.

Flujo sugerido: Login Admin → Crear patineta → Registrar cliente → Iniciar arriendo → BFF Dashboard → Finalizar → Reseña.

7 · Demo en Vivo

Flujo de Demostración (Docker)

Los 13 contenedores corren localmente con Docker: Gateway → BFF → 10 microservicios + PostgreSQL.

1Registro de Usuario — 200 OK

POST /api/auth/register → crea cuenta con rol CLIENT.

2Login — 200 OK

POST /api/auth/login → devuelve el token JWT de seguridad.

3Sin Credenciales — 403 Forbidden

GET a endpoint protegido sin token → el sistema rechaza la petición.

4Operación del Negocio — 201 Created

POST /api/rentals/start con token válido → arriendo creado con éxito.

El paso 3 (403 sin token) demuestra que la seguridad JWT protege el endpoint core antes de la operación válida del paso 4.

Conclusión

VeloUrbe — Arquitectura sólida y mantenible

La segregación en 10 microservicios, coordinada por un API Gateway sin código que enruta al BFF (patrón Gateway → BFF → servicios), produce un sistema cohesivo, desacoplado y tolerante a fallos.

Java 21 + Records inmutables, patrón CSR verificado, Flyway, JWT stateless, Swagger centralizado y una suite de pruebas automatizadas certifican el cumplimiento de los requerimientos técnicos y de calidad exigidos.
